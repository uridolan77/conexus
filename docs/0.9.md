
# Conexus v0.9 — Real Adapter Runtime Integration + Admin RBAC + Observability

## Goal

Turn the current BO/deployment lifecycle from a controlled UI around `conexus.adaptation` into a real Conexus runtime integration:

```text
1. BO admins have proper permissions, not all roles.
2. Conexus Gateway can register adapter profiles for runtime use.
3. Gateway requests can be associated with adapter profiles.
4. Observability data can be exposed back to adaptation for drift monitoring.
5. The BO can clearly distinguish “registered”, “canary”, “active”, and “real traffic-shaping” states.
```

## Current state

Already done in `conexus`:

```text
- BO proxy for adaptation plans/runs/profiles.
- BO proxy for publish / activate-canary / promote / rollback.
- Evaluation evidence viewer.
- Deployment events viewer.
- Idempotency-Key forwarding for deployment POSTs.
- Deployment success feedback with wasDuplicate / idempotent replay.
- Malformed JSON rejection before proxying deployment actions.
```

Remaining gap: Conexus still mostly treats adapter deployment as a BO/proxy concern. It does not yet have a true runtime adapter-profile registry or traffic-routing layer.

# Phase 1 — Admin RBAC for adaptation deployment actions

## Problem

The proxy currently maps every authenticated BO admin to:

```text
ComplianceReviewer
AdaptationPublisher
AdaptationOperator
```

That is fine for dev, but not production.

## Add admin permissions

Add permissions/capabilities such as:

```text
adaptation.view
adaptation.approve
adaptation.publish
adaptation.operate
adaptation.rollback
```

Suggested mapping:

```text
Admin / SuperAdmin:
  all adaptation permissions

AdaptationReviewer:
  adaptation.view
  adaptation.approve
  adaptation.rollback

AdaptationPublisher:
  adaptation.view
  adaptation.publish

AdaptationOperator:
  adaptation.view
  adaptation.operate
  adaptation.rollback

ReadOnly:
  adaptation.view
```

## Backend tasks

Add role/permission fields to admin users if not already present:

```text
admin_users.roles_json
```

or a simple association table later.

Update `_deployment_identity(admin)`:

```text
adaptation.approve   -> ComplianceReviewer
adaptation.publish   -> AdaptationPublisher
adaptation.operate   -> AdaptationOperator
adaptation.rollback  -> ComplianceReviewer or AdaptationOperator
```

Reject proxy calls before forwarding when the BO admin lacks local Conexus permission.

## Tests

```text
publish_without_permission_returns_403_before_proxy
activate_canary_without_permission_returns_403_before_proxy
promote_without_permission_returns_403_before_proxy
rollback_without_permission_returns_403_before_proxy
read_only_can_view_profiles_but_cannot_deploy
super_admin_gets_all_adaptation_roles
```

# Phase 3 — Runtime request association with adapter profiles

## Goal

Conexus gateway requests should be able to record which adapter profile influenced routing/prompting.

## Add fields to `gateway_requests`

```text
adapter_profile_id nullable
gateway_profile_id nullable
domain_key nullable
adaptation_mode nullable
```

## Request-time behavior

Do not force all gateway calls through adaptation yet.

Add optional headers/body metadata:

```text
X-Conexus-Domain-Key: gaming-crm
X-Conexus-Adapter-Profile-Id: gw-...
```

or body fields if your gateway schema already supports metadata.

Initial behavior:

```text
- If gatewayProfileId is supplied and registered, attach it to request log.
- If domainKey is supplied and active profile exists, attach active/canary profile.
- If none supplied, preserve existing gateway behavior.
```

Do **not** change model routing behavior yet unless you are ready.

## Tests

```text
gateway_request_with_gateway_profile_logs_profile_id
gateway_request_with_unknown_gateway_profile_returns_400_or_ignores_based_on_policy
gateway_request_without_profile_preserves_existing_behavior
```

Recommended policy:

```text
unknown explicit gatewayProfileId -> 400
missing gatewayProfileId -> normal behavior
```

# Phase 4 — Canary / active routing state in Conexus

## Goal

Conexus should know the currently active/canary profile per domain.

Add table:

```text
gateway_adapter_profile_activations
```

Fields:

```text
id
domain_key
gateway_profile_id
status
canary_percent
previous_gateway_profile_id
created_at
activated_at
promoted_at
rolled_back_at
metadata_json
```

Internal endpoints:

```http
POST /internal/adapter-profiles/{gatewayProfileId}/activate-canary
POST /internal/adapter-profiles/{gatewayProfileId}/promote
POST /internal/adapter-profiles/{gatewayProfileId}/rollback
GET  /internal/domains/{domainKey}/active-profile
```

This lets `conexus.adaptation` eventually control real gateway routing, not just its own internal state.

## Runtime canary behavior

For gateway calls with `domainKey`:

```text
if active profile exists and no canary:
  use active

if active + canary:
  deterministically bucket by project_id / api_key / request_id
  send canaryPercent to canary profile
  rest to active
```

Do not alter LLM output behavior yet. First just log selected profile.

Later, selected profile can influence:

* model profile
* prompt profile
* citation policy
* retrieval config
* safety policy

## Tests

```text
canary_bucket_selects_canary_for_expected_percentage
same_project_buckets_stably
promote_makes_canary_active
rollback_restores_previous_active
```

# Phase 5 — Observability endpoint for `conexus.adaptation` drift monitoring

## Goal

Prepare for `conexus.adaptation` v0.5 drift.

Add internal endpoint:

```http
GET /internal/adapter-profiles/{gatewayProfileId}/observability
```

Query:

```text
since
until
```

Response:

```json
{
  "gatewayProfileId": "gw-...",
  "windowStart": "...",
  "windowEnd": "...",
  "requestCount": 123,
  "errorRate": 0.01,
  "latencyP95Ms": 1200,
  "citationFailureRate": 0.04,
  "refusalRate": 0.03,
  "userNegativeFeedbackRate": 0.00,
  "costPerAnswer": 0.0021
}
```

Initial implementation can compute only what Conexus already knows:

```text
requestCount
errorRate
latencyP95Ms
costPerAnswer
```

Set unknown metrics to `null` or `0` with docs:

```text
citationFailureRate: null
refusalRate: null
userNegativeFeedbackRate: null
```

Better to return nullable than fake confidence.

## Add DB indexes

```text
gateway_requests(gateway_profile_id, created_at)
gateway_requests(domain_key, created_at)
```

## Tests


# Definition of done

This Conexus phase is done when:

```text
Conexus has internal adapter profile registry.
Internal registration is protected by API key.
Adapter profile registration is idempotent.
Gateway requests can be associated with gateway profiles.
Conexus can store/query active/canary profile state per domain.
Conexus exposes observability snapshots per gateway profile.
BO distinguishes adaptation lifecycle from gateway runtime state.
Docs define the contract for conexus.adaptation to call Conexus.
Existing gateway behavior is preserved when no profile/domain is supplied.
All backend/frontend tests pass.
```
