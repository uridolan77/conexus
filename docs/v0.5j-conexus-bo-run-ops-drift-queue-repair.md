# Conexus Plan — BO Proxy + UI for Conexus.Adaptation v0.5 Operations

Repository: `uridolan77/conexus`

## Goal

Expose the new `conexus.adaptation` v0.5 operational features through the Conexus Back Office, while preserving the correct trust boundary:

```text
Browser → Conexus BO/backend proxy → conexus.adaptation
```

The browser must never call the adaptation service directly and must never send authoritative user IDs or roles.

This phase adds BO support for:

- run cancellation
- run retry
- run resume
- drift status
- check drift
- queue diagnostics
- queue repair dry-run
- queue repair apply

---

## Current Context

`conexus` already proxies and displays:

- adaptation plans
- adaptation runs
- run manifests
- evaluation evidence
- adapter profiles
- publish / activate canary / promote / rollback
- deployment events
- active profile by domain
- runtime state from Conexus gateway registry
- deployment idempotency-key forwarding

`conexus.adaptation` now exposes:

### Run operations

```http
POST /adaptation-runs/{id}/cancel
POST /adaptation-runs/{id}/retry
POST /adaptation-runs/{id}/resume
```

### Drift operations

```http
POST /adapter-profiles/{id}/check-drift
GET  /adapter-profiles/{id}/drift-status
```

### Queue diagnostics / repair

```http
GET  /adaptation-runs/queue/diagnostics
POST /adaptation-runs/queue/repair/dry-run
POST /adaptation-runs/queue/repair
```

---

## Non-goals

Do **not** implement:

- changes to `conexus.adaptation`
- direct browser calls to the adaptation service
- sending browser-supplied user IDs or roles
- new auth system
- real gateway prompt/model adaptation
- Conexus-side execution of adaptation runs
- automatic drift rollback UI
- changes to `/v1/chat/completions` behavior
- Hangfire or background workers in Conexus

## Phase A — Backend Proxy: Run Operations

Update:

```text

```

### A1. Cancel Run

Conexus route:

```http
POST /admin/adaptation/runs/{run_id}/cancel
```

Upstream:

```http
POST /adaptation-runs/{id}/cancel
```

Browser body:

```json
{
  "reason": "operator cancelled from BO"
}
```

Proxy body:

```json
{
  "cancelledByUserId": "<admin identity>",
  "reason": "operator cancelled from BO"
}
```

Permission:

```text
adaptation.operate
```

Validation:

- JSON body may be empty
- `reason` optional
- trim reason if provided
- ignore browser-supplied `cancelledByUserId`


```json
  "workItemId": null,
```

```json
  "requestedByUserId": "<admin identity>",

  "runId": null,
  "workItemId": null,
  "reason": "post-deploy check",
```

## Phase G — Run Detail UI

Update:

```text
frontend/app/adaptation/runs/[id]/page.tsx
```

Add `RunOperationsPanel`.

### Actions by Status

| Run status | UI actions |
|---|---|
| `Queued` | Cancel |
| `Running` | Cancel |
| `Failed` | Retry, Resume |
| `Completed` | none |
| `Cancelled` | optional Retry if supported; otherwise none |

### Cancel

- confirmation required
- optional reason
- call `cancelRun`
- refresh run detail

### Retry

- confirmation required
- call `retryRun`
- on success show new run link
- do not automatically navigate unless UX already does this elsewhere

### Resume

- confirmation required
- call `resumeRun`
- refresh current run
- if 409, show ProblemDetails with copy:
  - “Resume is conservative. Use Retry if resume is rejected.”
